MCP Dart Ecosystem

Полный план разработки: пакеты, стэк, развёртывание, воркеры — от нуля до работающего стэка за один день.

Принципы

  • Строгий MCP протокол — промышленная совместимость
  • Каждый компонент — отдельный pub-пакет Dart
  • aq_schema — единственный источник правды (JSON схемы + Dart интерфейсы)
  • Adapter stateless → горизонтально масштабируемый
  • Worker wrapper обязателен — единый контракт для любого языка
  • Auth — заглушка сейчас, реальный сервис позже без изменения архитектуры
  • Docker Compose — один файл, весь стэк поднят

Ограничения MCP (важно)

MCP не поддерживает нативный async/polling. Долгие задачи решаем через job_id pattern: tools/call возвращает job_id, отдельный tool get_job_status проверяет статус. Это de-facto стандарт в реальных реализациях.
  • Sync: adapter блокируется, ждёт Redis (до N секунд)
  • Async: сразу возвращает job_id + tool get_job_status
  • Streaming: фаза 2 (SSE / chunked)

aq_schema
ЯДРО / Источник правды
JSON Schema файлы + сгенерированные Dart классы. Все остальные пакеты зависят только от него. Никогда не зависит от других пакетов экосистемы.
deps: dart:core, json_annotation
mcp/ worker/ queue/ auth/
aq_mcp_core
MCP Protocol Layer
Полная реализация MCP протокола: JSON-RPC codec, все типы (Initialize, ToolsList, ToolCall, Error), lifecycle manager, STDIO transport, HTTP transport.
deps: aq_schema
aq_mcp_adapter
Adapter / Оркестратор
Главный сервер. Принимает MCP, агрегирует tools со всех воркеров, диспетчеризует в очередь, возвращает результат. Stateless, масштабируется горизонтально.
deps: aq_schema, aq_mcp_core, aq_queue, aq_auth
aq_worker
Dart Worker Base
Базовый класс для написания Dart воркеров. Включает: wrapper, auto-registration, health check loop, JSON schema validation, error normalization, timeout.
deps: aq_schema, aq_queue
aq_queue
Queue Abstraction
Абстракция над Redis. Job enqueue/dequeue, result store, pub-sub. Позволяет заменить Redis на другой брокер без изменения остального кода.
deps: aq_schema, redis (pub.dev)
aq_auth
Auth Middleware
Заглушка v1: пропускает всё, логирует. Интерфейс AuthProvider заготовлен — позже подключается реальный сервис (JWT, OAuth) без изменения остальных пакетов.
deps: aq_schema

Dart интерфейсы (обязательные)

// Все пакеты реализуют эти интерфейсы из aq_schema

abstract class McpTool {
  String get name;
  String get description;
  Map<String, dynamic> get inputSchema;
  AuthRequirement? get auth;
}

abstract class WorkerJob {
  String get jobId;
  String get tool;
  Map<String, dynamic> get payload;
  AuthContext? get auth;
  JobMeta? get meta;
}

abstract class WorkerResult {
  String get jobId;
  JobStatus get status;       // pending | running | done | failed
  Map<String, dynamic>? get result;
  String? get error;
}
aq_mcp_adapter — детали

Request Flow

01 MCP Client → JSON-RPC request (initialize / tools/list / tools/call)
02 Transport Layer → STDIO или HTTP handler парсит Content-Length frame
03 Auth Middleware → проверяет token (v1: mock, всегда OK)
04 MCP Handler → routes по method: initialize | tools/list | tools/call | get_job_status
05 tools/list → Worker Registry → агрегация capabilities → ответ клиенту
06 tools/call → Queue Dispatcher → генерирует job_id → кладёт в Redis
07 Mode check: sync → await Redis result (timeout) | async → сразу возвращает job_id
08 Serialize → MCP Response → клиент

Worker Registry

// In-memory + Redis persistence
class WorkerRegistry {
  // Регистрация воркера
  Future<void> register(WorkerRegistration reg);

  // Агрегация всех tools
  List<McpTool> allTools();

  // Найти воркер по tool name
  WorkerInfo? findWorker(String toolName);

  // Health check loop (каждые 30s)
  void startHealthCheck();

  // Деregistрация мёртвого воркера
  void evict(String workerId);
}

Python Worker (aq_py_wrapper)

# aq_py_wrapper — pip пакет
# Обёртывает ЛЮБОЙ готовый Python MCP сервер

from aq_py_wrapper import WorkerWrapper

# Подключаем любой MCP сервер из
# pip install mcp-server-filesystem
# pip install mcp-server-github
# и т.д.

wrapper = WorkerWrapper(
  worker_id="filesystem-worker",
  redis_url=os.environ["REDIS_URL"],
  adapter_url=os.environ["ADAPTER_URL"],
  tools=[
    {
      "name": "read_file",
      "description": "Read file from filesystem",
      "inputSchema": { ... }
    }
  ]
)

async def execute(tool, payload, auth):
  # вызываем внутренний MCP сервер
  return await mcp_server.call_tool(tool, payload)

wrapper.run(execute)

Deploy — Docker Compose
Один файл docker-compose.yml поднимает весь базовый стэк. Воркеры запускаются отдельно или через отдельные compose файлы.

docker-compose.yml (базовый стэк)

version: '3.9'
services:
  nginx:
    image: nginx:alpine
    ports: ["443:443", "80:80"]
    volumes: ["./nginx/nginx.conf:/etc/nginx/nginx.conf"]
    depends_on: [adapter]

  adapter:
    build: "./packages/aq_mcp_adapter"
    env_file: .env
    environment:
      REDIS_URL: redis://redis:6379
      MCP_PORT: 8080
      AUTH_ENABLED: "false"
    depends_on: [redis]
    restart: unless-stopped

  redis:
    image: redis:7-alpine
    command: redis-server --maxmemory 256mb --maxmemory-policy allkeys-lru
    volumes: ["redis_data:/data"]
    restart: unless-stopped

volumes:
  redis_data:

Nginx конфиг (MCP over HTTP)

# nginx/nginx.conf
upstream adapter {
  server adapter:8080;
  keepalive 32;
}

server {
  listen 80;
  location /mcp {
    proxy_pass http://adapter;
    proxy_http_version 1.1;
    proxy_set_header Connection "";
    proxy_read_timeout 300s;        # для долгих задач
    proxy_buffering off;
  }

  location /health {
    proxy_pass http://adapter/health;
  }
}

Фазы — план на сегодня и далее
Цель сегодня: работающий стэк с Python MCP воркером (filesystem или brave-search), подключённый к локальному агенту.
PHASE 0 aq_schema — ядро ~2ч
  • Создать репо, настроить монорепо структуру
  • aq_schema/mcp/schemas/ — JSON схемы initialize, tools_list, tools_call, error
  • aq_schema/worker/schemas/ — register, job, job_result, health
  • aq_schema/queue/schemas/ — queue_job, queue_result
  • Написать Dart модели (McpTool, WorkerJob, WorkerResult, AuthContext)
  • Написать валидаторы (jsonschema)
  • Тесты на схемы
PHASE 1 aq_queue + aq_auth ~1ч
  • aq_queue: Redis connection, enqueue job, dequeue, get result, set result
  • aq_queue: простой BLPOP loop для воркеров
  • aq_auth: интерфейс AuthProvider + MockAuthProvider (всегда OK)
  • Тесты с реальным Redis (testcontainers или docker)
PHASE 2 aq_mcp_core ~2ч
  • JSON-RPC codec (encode/decode)
  • STDIO transport (Content-Length framing)
  • HTTP transport (shelf или dart:io)
  • MCP lifecycle: initialize handshake
  • Handlers: tools/list, tools/call, get_job_status
  • Error mapping (worker errors → MCP errors)
PHASE 3 aq_mcp_adapter (MVP) ~2ч
  • Worker Registry (in-memory, сохранение в Redis)
  • Worker registration HTTP endpoint (POST /workers/register)
  • Health check loop + eviction
  • Queue Dispatcher: tools/call → Redis → await result
  • Capability aggregation → tools/list
  • Sync mode (await result) + Async mode (return job_id)
  • Dockerfile + health endpoint
PHASE 4 Python Worker + Docker Compose ~1ч
  • aq_py_wrapper (pip пакет): registration, Redis BLPOP, health check
  • Пример воркера: mcp-server-filesystem или mcp-server-brave-search
  • docker-compose.yml: nginx + adapter + redis
  • docker-compose.workers-python.yml: python воркер
  • .env.example для всех сервисов
PHASE 5 Интеграция и тест ~1ч
  • docker compose up → стэк поднят
  • Тест tools/list — видны tools из python воркера
  • Тест tools/call — выполняется через Redis → Python → результат
  • Подключение к Claude Desktop или другому агенту
  • E2E тест: агент вызывает tool, получает результат

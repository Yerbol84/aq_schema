# ТЗ: Append-only логи для WorkflowRun

**От:** aq_graph_engine  
**Кому:** aq_data_layer  
**Приоритет:** MEDIUM  
**Тип:** Оптимизация + новая функциональность

---

## Контекст

При выполнении графа каждый шаг генерирует строку лога. Сейчас логи хранятся как JSON-массив в поле `logsJson: String` модели `WorkflowRun`.

Запись лога сейчас:
```
1. Прочитать WorkflowRun из БД
2. Распарсить logsJson → List<String>
3. Добавить новую строку
4. Сериализовать обратно в JSON
5. Сохранить WorkflowRun в БД
```

Это **read-modify-write** на каждую строку лога.

---

## Проблема

### Производительность
При 100 узлах в графе: 100 операций read-modify-write.  
При параллельных ветках: race condition — два узла читают одно и то же `logsJson`, каждый добавляет свою строку, второй перезаписывает первого.

### Масштабируемость
Чем длиннее граф — тем больше нагрузка на БД.  
Логи растут линейно, но нагрузка на запись — тоже линейная (каждый узел = write).

---

## Что нужно

Механизм **атомарного добавления** одной строки к логам `WorkflowRun` без read-modify-write.

В `IRunRepository` уже есть метод (сейчас no-op):
```dart
Future<void> appendLog(String runId, String entry);
```

Нужна реализация которая:
1. Атомарно добавляет строку к логам без чтения всего списка
2. Не создаёт race condition при параллельных вызовах
3. Не требует блокировки всей записи `WorkflowRun`

---

## Требования

### Функциональные

- `appendLog(runId, entry)` — добавить одну строку к логам run
- Атомарность — параллельные вызовы не теряют строки
- Порядок — строки добавляются в порядке вызовов (best effort, не строгий)
- `getRun(runId).logsJson` — должен возвращать все добавленные строки

### Нефункциональные

- Операция должна быть значительно быстрее текущего read-modify-write
- Не должна блокировать другие операции с `WorkflowRun` (статус, contextJson)

---

## Варианты реализации (на усмотрение команды)

Возможные подходы (не ограничиваться ими):
- Отдельная таблица `workflow_run_logs` с `INSERT` (true append)
- Postgres `array_append` в одном SQL запросе
- Другой механизм который даёт атомарный append

## Текущий workaround

`appendLog` — no-op дефолт в `IRunRepository`.  
Логи пишутся батчем через `updateRunLog` при смене статуса.  
Работает корректно, но не оптимально при длинных графах.
